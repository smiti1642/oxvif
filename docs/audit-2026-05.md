# oxvif implemented-operation fidelity audit — 2026-05-29

> Companion to the coverage map (oxdm `ODM.md`). Coverage = *which* ops exist; this
> audit = whether *implemented* ops send all required request params and parse all
> needed response fields, vs the ONVIF schema (`onvif.xsd` v25.12 / the per-service
> WSDLs). Method: read each `Set*`/`Get*`'s request body builder + the type's
> `from_xml`, check read-modify-write round-trip parity and element/cardinality
> conformance. Findings confirmed against the live `onvif.xsd` where the `docs/*.md`
> summaries defer type internals to the schema.

## Fixed in this pass

### `VideoEncoderConfiguration` (Media1) — `SetVideoEncoderConfiguration` was schema-invalid
Confirmed against `onvif.xsd` v25.12 the complexType sequence is
`Encoding, Resolution, Quality, RateControl[0..1], MPEG4[0..1], H264[0..1],
Multicast[1], SessionTimeout[1]`, with `GuaranteedFrameRate` as an **attribute**.
oxvif's `to_xml_body` violated this several ways — strict, schema-validating cameras
reject the whole Set (silent "change didn't apply"):

| # | Bug | Fix |
|---|-----|-----|
| 1 | `Quality` emitted **last** (after Multicast) | reordered: `Quality` right after `Resolution`, before `RateControl` |
| 2 | `SessionTimeout` `[1]` required — **never sent** | added `session_timeout: Option<String>` field, parsed in `from_xml`, emitted when present (preserved across read-modify-write) |
| 3 | `GuaranteedFrameRate` emitted/parsed as a **child element** | now emitted as an attribute; `from_xml` reads the attribute, falls back to the child form for lenient devices |

Files: `src/types/video.rs` (struct + `from_xml` + `to_xml_body`),
`src/tests/types_tests.rs` (round-trip test asserts ordering + SessionTimeout +
attribute form). Plausibly the real cause of the "sub-stream resolution change didn't
apply" report on stricter cameras.

### `PtzConfiguration` — `SetConfiguration` emitted malformed XML for optional fields
The `opt_str` helper in `to_xml_body` closed optional string elements as `<tt:/Tag>`
(slash misplaced) instead of `</tt:Tag>`. Any `ptz_set_configuration` carrying
`DefaultPTZTimeout` or a coordinate-space URI sent invalid XML → strict cameras reject
outright. Fixed the helper; added `test_ptz_configuration_to_xml_body_closes_optional_tags`.
File: `src/types/ptz_config.rs`.

> Verification note: an adjacent **false positive** — `</tptz:SetConfiguration>` in
> `client/ptz.rs` rendered as `<\…>` in Grep output — was confirmed correct via Read and
> left alone. Confirm XML findings with Read, not Grep's content display.

## Audited — clean

- **`VideoSourceConfiguration`** / **`ImagingSettings`** — full round-trip parity;
  `ForcePersistence` + correct tokens sent.
- **`OsdConfiguration`** (`set_osd` / `create_osd`) — well-formed, namespace-correct
  (`trt:` element, `tt:` children); round-trips token / vsc / type / position / text / image.
- **`PtzConfiguration`** request envelope (`SetConfiguration` + `ForcePersistence`) —
  well-formed (the optional-field bug above lived inside `to_xml_body`, now fixed).
- **Device-service writes** — `set_network_interfaces`, `set_dns`, `set_ntp`,
  `set_network_default_gateway`, `set_network_protocols`, `set_hostname`,
  `set_system_date_and_time` (element order matches the XSD), `create_users` / `set_user` /
  `delete_users`, `set_relay_output_state` / `set_relay_output_settings`. All structurally
  correct.
- **Media2 `VideoEncoderConfiguration2`** — well-formed; the `tr2` element ordering was not
  deeply verified against the Media2 schema (Media2 is lenient; oxdm writes via Media1).

## Flagged — not fixed (need a decision / lower priority)

- **`VideoEncoderConfiguration` H265 in Media1**: oxvif emits `<tt:H265>`, which is **not
  in the Media1 schema** (Media1 = JPEG/MPEG4/H264 only; H265 is Media2). Left in place
  defensively (some vendors accept it via Media1), but a strict camera rejects it. Proper
  H265 path is Media2 — oxdm currently writes via Media1. *Decision needed:* route H265
  encoders through Media2 in oxdm, or drop the H265 element from the Media1 body.
- **`ImagingSettings` model depth**: only Mode-level fields are modeled. Manual
  `Exposure` (ExposureTime / Gain / Iris / Priority), `WhiteBalance` Cr/Cb gain, and
  `Focus` Near/FarLimit are **not** parsed or sent — can't drive fully-manual exposure/WB
  via oxvif. Model-completeness gap, not a round-trip bug.
- **`set_network_interfaces` / `set_network_default_gateway`**: IPv4-only; IPv6, MTU, and
  multiple manual addresses are not sent. Incompleteness, fine for current oxdm use.
- **`set_ntp`**: a manual server is always sent as `Type=DNS` + `DNSname`, even when the
  caller passes an IP literal (should be `Type=IPv4` + `IPv4Address`). Most devices tolerate.
