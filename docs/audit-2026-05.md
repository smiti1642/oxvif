# oxvif implemented-operation fidelity audit — 2026-05-29

> **2026-06-04 update — 0.9.8 cleanup.** The three "Flagged — not fixed" items
> from this audit are now resolved. See "Fixed in 0.9.8" below.

> Companion to the coverage map (oxdm `ODM.md`). Coverage = *which* ops exist; this
> audit = whether *implemented* ops send all required request params and parse all
> needed response fields, vs the ONVIF schema (`onvif.xsd` v25.12 / the per-service
> WSDLs). Method: read each `Set*`/`Get*`'s request body builder + the type's
> `from_xml`, check read-modify-write round-trip parity and element/cardinality
> conformance. Findings confirmed against the live `onvif.xsd` where the `docs/*.md`
> summaries defer type internals to the schema.

## Fixed in this pass

### `VideoEncoderConfiguration` (Media1) — `SetVideoEncoderConfiguration` was schema-invalid
Confirmed against `onvif.xsd` v25.12 the complexType sequence is
`Encoding, Resolution, Quality, RateControl[0..1], MPEG4[0..1], H264[0..1],
Multicast[1], SessionTimeout[1]`, with `GuaranteedFrameRate` as an **attribute**.
oxvif's `to_xml_body` violated this several ways — strict, schema-validating cameras
reject the whole Set (silent "change didn't apply"):

| # | Bug | Fix |
|---|-----|-----|
| 1 | `Quality` emitted **last** (after Multicast) | reordered: `Quality` right after `Resolution`, before `RateControl` |
| 2 | `SessionTimeout` `[1]` required — **never sent** | added `session_timeout: Option<String>` field, parsed in `from_xml`, emitted when present (preserved across read-modify-write) |
| 3 | `GuaranteedFrameRate` emitted/parsed as a **child element** | now emitted as an attribute; `from_xml` reads the attribute, falls back to the child form for lenient devices |

Files: `src/types/video.rs` (struct + `from_xml` + `to_xml_body`),
`src/tests/types_tests.rs` (round-trip test asserts ordering + SessionTimeout +
attribute form). Plausibly the real cause of the "sub-stream resolution change didn't
apply" report on stricter cameras.

### `PtzConfiguration` — `SetConfiguration` emitted malformed XML for optional fields
The `opt_str` helper in `to_xml_body` closed optional string elements as `<tt:/Tag>`
(slash misplaced) instead of `</tt:Tag>`. Any `ptz_set_configuration` carrying
`DefaultPTZTimeout` or a coordinate-space URI sent invalid XML → strict cameras reject
outright. Fixed the helper; added `test_ptz_configuration_to_xml_body_closes_optional_tags`.
File: `src/types/ptz_config.rs`.

> Verification note: an adjacent **false positive** — `</tptz:SetConfiguration>` in
> `client/ptz.rs` rendered as `<\…>` in Grep output — was confirmed correct via Read and
> left alone. Confirm XML findings with Read, not Grep's content display.

## Fixed in 0.9.8

### `VideoEncoderConfiguration` H265 in Media1 — gated at the client API
Media1 schema is JPEG / MPEG4 / H264 only; H265 belongs to Media2. The Media1
[`OnvifClient::set_video_encoder_configuration`](../src/client/media.rs) now
returns `OnvifError::InvalidArgument` with a clear hint (use
`set_video_encoder_configuration_media2`) when the caller passes
`encoding == VideoEncoding::H265`, **without** sending the request. The
serializer itself is unchanged (still emits `<tt:H265>` if asked), so the
gate lives at the API boundary, not deep in the type.

### `ImagingSettings` model depth — manual exposure / WB / focus now reachable
Added eight `Option<...>` fields to `ImagingSettings`:
`exposure_time`, `exposure_gain`, `exposure_iris`, `exposure_priority`,
`wb_cr_gain`, `wb_cb_gain`, `focus_near_limit`, `focus_far_limit`.
`from_xml` parses them via path lookups; `to_xml_body` emits them in the
correct intra-block XSD sequence (Exposure20: `Mode, Priority, ExposureTime,
Gain, Iris`; WhiteBalance20: `Mode, CrGain, CbGain`; FocusConfiguration20:
`AutoFocusMode, DefaultSpeed, NearLimit, FarLimit`).

### `set_network_interfaces` — IPv6 + MTU + multiple manual addresses
Replaced the positional API `(token, enabled, ipv4_address, prefix_length,
from_dhcp)` with a struct-based one taking `&NetworkInterfaceConfig`,
which carries optional `mtu`, optional IPv4 stack (`IpStackConfig`), and
optional IPv6 stack. Each stack supports any number of `ManualAddress`
entries. IPv6 DHCP is exposed as a bool (`from_dhcp` → `Stateful` / `Off`);
the four-value enum is left for a later release if a real caller needs it.
This is a **breaking API change** — callers (oxdm) need to migrate.

## Flagged — not fixed (need a decision / lower priority)

- **`set_network_default_gateway`**: IPv4-only. Not addressed in 0.9.8 — same
  surface as `set_network_interfaces` but a separate WSDL call; will roll up
  when an IPv6 gateway is actually needed downstream.
- **`set_ntp`**: a manual server is always sent as `Type=DNS` + `DNSname`, even when the
  caller passes an IP literal (should be `Type=IPv4` + `IPv4Address`). Most devices tolerate.

## Audited — clean (read paths)

- **`get_profiles` / `MediaProfile`** — surfaces video source/encoder, audio
  source/encoder, PTZ, and video-source tokens. Only a bound *metadata*-config token is
  not surfaced (minor; oxdm doesn't use it).
- **`get_network_interfaces` / `NetworkInterface`** — parses IPv4 manual+DHCP
  address/prefix and IPv6 enabled/DHCP/manual/link-local. Asymmetry worth noting: read
  surfaces IPv6 but `set_network_interfaces` only writes IPv4 (see Flagged).
- **`get_capabilities` / `Capabilities`** — parses every service URL
  (Device/Media/Events/Analytics/PTZ/Imaging/Recording/Search/Replay/DeviceIO/Media2) plus
  system flags (`firmware_upgrade` / `system_backup`). Drives session routing + Tier-B
  gating. Solid. Nuance: session build uses `GetCapabilities`; a device that only
  advertises Media2 via `GetServices` would report no Media2 URL here.

## Not yet audited (low oxdm relevance)

Recording / search / replay read getters; Media2 `VideoEncoderConfiguration2` element
order vs the `tr2` schema. Both are paths oxdm does not currently surface.
